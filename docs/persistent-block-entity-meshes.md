# Persistent block entity meshes

## Scope

Optiminium captures audited vanilla block-entity model output once, uploads it to static GPU vertex buffers, and shares it by visual key. It does not cache world position, light, overlay, sign text, or animation transforms in geometry.

Supported paths:

- decorated pots while not wobbling;
- stationary bells, with shaking bells falling back for animation;
- fully closed single, trapped, and ender chests;
- the static bottom of an opening single, trapped, or ender chest while the lid and lock remain vanilla-rendered;
- standing, wall, and hanging sign boards while both text faces remain vanilla-rendered.

Double chests remain vanilla because their renderer combines neighbor openness and lighting. Furnace, blast-furnace, smoker, barrel, and hopper geometry is already part of Minecraft's chunk mesh and is not generated by a block-entity renderer.

Only the exact vanilla renderer classes are accepted. Replacement or modded renderers, sorted/translucent render types, unsupported block entities, double chests, and active pot wobble use vanilla rendering.

## Architecture

Geometry keys contain the block-entity type, block-state ID, geometry/material variant, and renderer class. Decorated-pot sherd layouts are part of the variant. Sign text is deliberately absent.

Vanilla model output is captured with an identity pose. A custom shader supplies world/camera transform, packed light, and overlay from a reusable instance buffer. Normals are transformed per instance. Static geometry uses `VertexBuffer.Usage.STATIC`; instance storage is allocated once per cached mesh, grows geometrically, and is updated once for each visible mesh batch.

Visible instances are queued during block-entity enumeration and flushed once after the block-entity pass. One instanced draw is emitted per mesh render layer, rather than per block entity. Instance data uses a preallocated three-buffer GPU ring so an update does not overwrite storage still consumed by a recent frame. A visual key must have appeared at least 128 times in the preceding frame before persistent rendering engages. Sparse and medium-sized scenes stay on Minecraft's batched vanilla path. Development runs can override this with `-Poptiminium.persistentMeshMinInstances=N`.

Failed culling only omits that frame's instance. It does not delete the shared mesh. Resource/model/texture reload and world unload close both geometry and instance buffers on the render thread. Block-state, chest connection, facing, wood/material, and pot-decoration changes naturally select a different key. Sign text changes continue through the uncached vanilla text path immediately.

## Diagnostics

Enable `blockEntityRenderCacheDebug` for the overlay and one-second log samples. Metrics include:

- cached meshes, hits, misses, and rebuilds per second;
- static geometry and per-frame instance-buffer uploads;
- cached, culled, and vanilla-fallback instances per frame;
- vertices avoided and instanced draw calls per frame;
- total allocated GPU geometry plus instance-buffer capacity;
- client FPS and CPU frame time.

Nonessential counters and timing are gated when both cache debug and the render profiler are disabled.

## Runtime evidence

Tests use NeoForge 21.1.190 / Minecraft 1.21.1 and the local Hermitcraft 9 save.

Before instancing, a small view with nine supported instances exposed a regression: cache off averaged 97.5 FPS and 10.53 ms CPU frame time, while immediate per-instance VBO drawing averaged 92.3 FPS and 11.83 ms. This result caused the immediate renderer to be replaced, not accepted as success.

The forced instancing validation (`persistentMeshMinInstances=1`) subsequently exercised:

- 23 shared geometry meshes and 23 one-time geometry uploads;
- up to 32 cached instances in sampled frames;
- up to 1,320 CPU-generated vertices avoided per sampled frame;
- 18-19 instanced draws for those instances because they spanned many distinct state/texture keys;
- 41,792 bytes of combined geometry and instance-buffer capacity;
- clean shader, mixin, OpenGL, resource-reload, world-load, and world-unload logs.

An isolated copied-world benchmark established the density crossover. The first version, which updated one instance buffer every frame, stalled and regressed. A preallocated three-buffer upload ring fixed that synchronization hazard.

With 64 identical decorated pots, vanilla remained faster: its final-ten-sample median was 343.5 FPS / 2.791 ms versus 305 FPS / 3.056 ms for persistent instancing. This is why the production density gate is higher than 64.

With 256 identical decorated pots and the same persisted spectator camera:

- vanilla final-ten average: 273.2 FPS and 4.406 ms CPU frame time;
- persistent final-ten average: 400.3 FPS and 2.530 ms CPU frame time;
- improvement: 46.5% average FPS and 42.6% lower average CPU frame time;
- 28,672 CPU-generated vertices avoided per sampled frame;
- one persistent instance upload and one instanced draw per sampled frame;
- one geometry mesh/upload and 102,336 bytes of total geometry plus ring-buffer GPU capacity.

The first screenshot comparison caught and rejected an instancing bug: the custom draw omitted Minecraft's global model-view matrix, causing submitted instances to disappear. After applying the same `RenderSystem.getModelViewMatrix()` stage used by vanilla `BufferUploader`, deterministic GUI-free cache-off/on captures at 854x480 measured:

- 0.0405% different pixels over the complete frame;
- 0.00022 mean absolute RGB-channel delta over the complete frame;
- 19,940 exactly equal pixels in the measured pot region;
- 151 pot-region pixels differing by exactly one value in one RGB channel;
- no larger pot-region difference and no geometry, transform, texture, lighting, fog, or depth mismatch.

The paired images are written by the opt-in benchmark screenshot property and use a pinned client/server camera, fixed time, and hidden GUI.

The production threshold is therefore 128: 64-instance groups retain vanilla batching, while large groups such as the measured 256-instance case use persistent instancing.
